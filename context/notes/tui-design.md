# TUI Design Decisions

Rationale and principles behind TUI design choices. Implementation state is documented in `systems/tui.md`; this note captures the **why** behind the choices.

---

## Core design principles

1. **Dynamic over hardcoded.** Bars, blocks, and layouts size to percentages of their container, never fixed character counts. Think `Percentage(80)` not `Length(25)`. This adapts to any terminal size and makes adding new content trivial.

2. **Density over whitespace.** Empty space is wasted. If a block has leftover room, fill it with content. If content is shorter than its container, shrink the container.

3. **Mouse-first, keyboard-enhanced.** Mouse and touchpad should be completely natural. Keyboard shortcuts are power-user accelerators.

4. **Grade is the primary metric.** `evaluation_status` (strong_fit/weak_fit/no_fit) is just a coarser bucketing of `grade`. Display grade only, never evaluation_status.

---

## Why bar charts over pie charts

Terminal pie charts (braille circles) look like blobs. Rectangular cells distort circles, small slices become unreadable, and you can't label segments. Bar charts are the strongest terminal visualisation — labelled, coloured, proportional, readable at any size.

---

## Responsive layout strategy

- **Full mode** (120+ columns): side-by-side master/detail, full stats blocks, wide bars
- **Compact mode** (<100 columns): single-column layout, list above detail
- **Minimal mode** (<80 columns): abbreviated labels, no detail panel

Implemented in TUI v4. Three modes active based on terminal width: full (120+), stacked (<100), compact (<80). The `widgets/layout.rs` module provides a `distribute()` function for dynamic block sizing.

## Fit data simplification

`evaluation_status` adds no information beyond `grade`. Remove from display everywhere. Future possibility: split into **role quality** and **accessibility** as separate dimensions. Would require schema change.

## Scroll behaviour

Mouse/touchpad scroll should move the viewport (offset), not jump through entries. The selection stays in place. Momentum scrolling on MacBook trackpads should be batched to prevent flying through hundreds of entries. Mouse scroll targets whichever pane the cursor is over, regardless of keyboard focus.

## Modularisation rationale (session 7)

Three monolithic files were split into directories when they exceeded ~500 lines and contained clearly separable concerns:

| File | Lines | Problem | Split |
|------|-------|---------|-------|
| `app.rs` | 1,112 | State definition, navigation logic, user actions, pipeline management, and cleanup all interleaved | `app/` — 6 files by concern (state, navigation, actions, pipeline, cleanup, mod) |
| `handler.rs` | 490 | Key navigation, overlay input, and mouse events share no state but were in one dispatch function | `handler/` — 4 files by event source (navigation keys, overlay keys, mouse) |
| `views/mod.rs` | 438 | Chrome (tabs, status bar), overlays (help, search, grade picker), and draw dispatch mixed together | `mod.rs` + `chrome.rs` + `overlays.rs` by rendering responsibility |

The threshold is pragmatic, not dogmatic: files under ~300 lines that handle one concern stay as single files (theme.rs, queries.rs). The split preserves the `App` struct as the shared state — modules are methods on `App`, not separate types.

## Component architecture patterns (session 7)

### One-line kanban cards

Pipeline cards reduced from multi-line blocks to single lines: `[A] Software Engineer — Citadel`. Rationale: the kanban view's purpose is overview and triage, not detail reading. Detail is available via quick-peek (Space) or drill-down (Enter). Fitting more cards per column means less scrolling and better at-a-glance coverage.

### Activity heatmap

GitHub-style 7×12 grid (days of week × weeks) showing daily activity density. Each cell coloured by the number of events (discoveries, searches, gradings, decisions) on that date. Purpose: visual pulse of project momentum — gaps in the heatmap surface when the job search is going cold.

### Quick-peek popup

Space key opens a lightweight detail overlay without leaving the current list position. Implemented as a centred popup rendered on top of the current view. Dismissed by Space or Esc. This avoids the context-switch cost of Enter (which drill into detail and changes focus). The pattern: list browsing should be fast, detail should be available without losing your place.

### Breadcrumb navigation trail

A single line below the tab bar showing the navigation path: `Dashboard > Companies > Citadel Securities > Jobs`. Updates dynamically as the user drills down. Provides orientation in nested views and makes the back-navigation path visible.

### Smart job grouping (Ctrl+G)

Toggle that groups the jobs list by company instead of flat sorted-by-grade. When active, jobs are visually grouped under company headers. Useful when evaluating all roles at a specific company or spotting companies with multiple opportunities.

### Contextual status bar

The status bar's right side changes based on the active view: job count in Jobs view, company count in Companies view, pipeline stats in Pipeline view. The session timer (HH:MM:SS since TUI launch) appears in the far right of all views.

## Focus mode (session 7 fix)

The `f` key now correctly hides F-graded jobs, C-graded jobs, AND applied jobs from the list. Previously it only hid F/C grades. Applied jobs are noise once acted on — they belong in the Pipeline view, not cluttering the Jobs list during evaluation.
